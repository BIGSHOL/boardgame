# PRD (제품 요구사항 정의서)

> Hanyang: The Foundation - 조선 초기 한양 건설 전략 보드게임

---

## MVP 캡슐

| # | 항목 | 내용 |
|---|------|------|
| 1 | 목표 | 보드게임 프로토타입을 디지털로 테스트하여 실제 제작 전 밸런싱 검증 |
| 2 | 페르소나 | 보드게임 동호회 회원 및 게임 디자이너 본인 |
| 3 | 핵심 기능 | FEAT-2: 워커 파견 및 자원 관리 (전략적 선택의 핵심) |
| 4 | 성공 지표 (노스스타) | 동호회원 5명이 완전한 게임 1판 테스트 완료 |

## 1. 문제 정의

### 1.1 현재 상황 (As-Is)
보드게임을 디자인할 때 실물 프로토타입을 만들어 테스트하는 것은 마치 요리사가 매번 전체 코스 요리를 만들어야 한 가지 재료를 바꿔볼 수 있는 것과 같습니다. 카드를 인쇄하고, 토큰을 만들고, 보드를 제작하는 데 많은 시간과 비용이 들며, 작은 수정도 처음부터 다시 만들어야 합니다.

### 1.2 목표 상황 (To-Be)
디지털 프로토타입으로 빠르게 테스트하고 수정할 수 있어, 마치 시뮬레이터에서 요리를 미리 맛보는 것처럼 실물 제작 전에 게임 밸런스를 검증합니다. 멀리 있는 테스터들과도 온라인으로 함께 플레이하며 피드백을 수집할 수 있습니다.

### 1.3 왜 지금인가? (Why Now)
- "한양: 도읍의 설계자들" 게임 기획이 완성된 상태
- 보드게임 제작 경험을 쌓고 싶은 동기
- 실제 크라우드펀딩 전 검증이 필요
- 온라인 보드게임 플랫폼의 대중화로 테스터 접근성 향상

## 2. 사용자 페르소나

### 2.1 주요 페르소나

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "디자이너 동현" |
| 나이/직업 | 30대 보드게임 디자이너 지망생 |
| 핵심 니즈 | 자신이 디자인한 게임을 빠르게 테스트하고 밸런싱 조정 |
| 불편 포인트 | 실물 프로토타입 제작의 시간/비용, 테스터 모집의 어려움 |
| 사용 상황 | 집에서 혼자 룰 테스트, 동호회원들과 온라인 테스트 세션 |

### 2.2 페르소나의 하루 (시나리오)
동현은 퇴근 후 자신이 디자인한 "한양" 보드게임의 새로운 아이디어를 테스트하고 싶습니다. 웹 브라우저를 열고 게임에 접속해 혼자서 4인 플레이를 시뮬레이션합니다. "배산임수 보너스가 너무 강한 것 같은데?" 생각하며 수치를 조정합니다.

주말에는 동호회 카톡방에 "오늘 저녁 8시에 한양 테스트 어때요?"라고 메시지를 보내고, 5명의 친구들이 각자 집에서 온라인으로 접속해 함께 플레이합니다. 게임이 끝나고 "관청 건물 비용이 너무 높아요", "풍수지리 보너스 설명이 헷갈려요" 같은 피드백을 수집합니다.

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩/로그인)
- [ ] 사용자로서, 간편하게 가입할 수 있어야 한다. 왜냐하면 빠르게 게임 테스트를 시작하고 싶기 때문이다.
- [ ] 사용자로서, 로그인 상태가 유지되어야 한다. 왜냐하면 진행 중인 게임에 재접속할 수 있어야 하기 때문이다.
- [ ] 사용자로서, 게임 로비를 만들고 친구를 초대할 수 있어야 한다. 왜냐하면 특정 멤버와 테스트하고 싶기 때문이다.

### FEAT-1: 게임 보드 및 타일 배치
- [ ] 플레이어로서, 메인 보드(한양 지도)를 볼 수 있어야 한다. 왜냐하면 건설 상황을 파악해야 하기 때문이다.
- [ ] 플레이어로서, 개인 보드(건축가의 책상)를 관리할 수 있어야 한다. 왜냐하면 자원과 워커를 관리해야 하기 때문이다.
- [ ] 플레이어로서, 건물 타일을 메인 보드에 배치할 수 있어야 한다. 왜냐하면 이것이 게임의 핵심 행동이기 때문이다.

### FEAT-2: 워커 파견 및 자원 관리 (MVP)
- [ ] 플레이어로서, 견습공을 개인 보드에 배치해 자원을 획득할 수 있어야 한다. 왜냐하면 건설에 필요한 재료가 필요하기 때문이다.
- [ ] 플레이어로서, 관료를 메인 보드로 파견할 수 있어야 한다. 왜냐하면 건물을 짓고 영향력을 높여야 하기 때문이다.
- [ ] 플레이어로서, 파견된 관료는 건물에 영구 배치됨을 이해해야 한다. 왜냐하면 이것이 "파견 vs 회수" 전략의 핵심이기 때문이다.
- [ ] 플레이어로서, 자원(목재, 석재, 기와, 먹)을 확인하고 사용할 수 있어야 한다. 왜냐하면 건설 비용을 지불해야 하기 때문이다.

### FEAT-3: 풍수지리 보너스 계산 (v1)
- [ ] 플레이어로서, 배산임수 조건 충족 시 보너스를 받을 수 있어야 한다. 왜냐하면 이것이 게임의 테마적 차별점이기 때문이다.
- [ ] 플레이어로서, 인접 건물 시너지를 확인할 수 있어야 한다. 왜냐하면 전략적 배치가 점수에 영향을 주기 때문이다.

## 4. 성공 지표

### 4.1 노스스타 지표 (결과 지표)
| 지표명 | 정의 | 목표치 | 측정 주기 |
|--------|------|--------|----------|
| 완료된 테스트 게임 수 | 시작부터 끝까지 완료된 게임 세션 | 10회 | 월간 |
| 참여 테스터 수 | 최소 1게임 이상 플레이한 사용자 | 5명 | 월간 |

### 4.2 입력 지표 (행동 지표)
| 지표명 | 정의 | 목표치 |
|--------|------|--------|
| 테스트 세션 수 | 시작된 게임 세션 총 수 | 주 3회 |
| 수집된 피드백 | 게임 후 피드백 코멘트 수 | 게임당 3개 |

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- FEAT-2: 워커 파견 및 자원 관리 (핵심 전략 메카닉)
- FEAT-1: 기본 게임 보드 및 타일 배치 (시각적 피드백)
- FEAT-0: 기본 로그인 및 게임 로비 (멀티플레이)
- 실시간 멀티플레이 (WebSocket)

### 5.2 옵션 (시간 허용 시)
- FEAT-3: 풍수지리 보너스 계산
- 카드 드래프팅 시스템 (청사진)
- 게임 리플레이 저장

### 5.3 Non-goals (명시적 제외)
- AI 상대 (싱글플레이) - 멀티플레이 테스트가 목적
- 랭킹/리더보드 - 프로토타입 단계에 불필요
- 모바일 네이티브 앱 - 반응형 웹으로 충분
- 결제/수익화 - 테스트 목적의 무료 서비스

## 7. 해결 가설과 실험

### 7.1 가설
> 만약 디지털 프로토타입으로 게임을 테스트하면, 실물 제작 전 밸런싱 문제를 발견하고 수정할 수 있을 것이다. 왜냐하면 빠른 반복 테스트와 원격 피드백 수집이 가능하기 때문이다.

### 7.2 실험 루프

```
실험 → 관측 → 학습 → 다음 가설
```

| 단계 | 내용 |
|------|------|
| 실험 | 동호회원 5명과 온라인 테스트 플레이 진행 |
| 관측 | 게임 시간, 승리 전략, 테스터 피드백 |
| 학습 | 밸런스 이슈 3개 이상 발견 시 성공 |
| 다음 | 발견된 이슈 수정 후 재테스트 |
